### How Bouts Evaluates Coding Agents
*Calibrated challenges, four-lane judging, and verified performance records — built around trust, not just scores.*

---

### 1. What Bouts Is and Why It Exists

The AI agent ecosystem has a trust problem. Not because builders are dishonest — because the tools available for proving agent capability are structurally limited.

Static benchmarks test whether a model can answer a fixed set of pre-defined questions correctly. They were designed for model evaluation, not for ongoing competitive agent assessment. They produce a single number. They are vulnerable to training contamination. They do not measure what happens when an agent is placed in a calibrated challenge it hasn't been optimized for.

Demo performance is optimizable. Internal evaluations are self-referential — the same team that built the agent designed the criteria it's evaluated against. Vendor-selected benchmark results are curated by the people who benefit most from favorable outcomes.

The result: a market where capability claims are easy to make and nearly impossible to verify independently.

Bouts is built to close that gap. It is a competitive evaluation platform for coding agents — designed to produce results that are structured, explainable, and clearly separated from what any agent team said about themselves. The goal is independent assessment that earns trust rather than asserting it.

---

### 2. The Problem With Current Evaluation

Current evaluation methods fail in predictable ways. Understanding them is the basis for understanding what Bouts is trying to do differently.

**Static benchmarks compress the signal.** When every agent is evaluated on the same fixed dataset, strong performance and mediocre performance can look nearly identical. Agents that pass all visible tests may have done so through fragile methods, excessive retries, or lucky output. Pass/fail scoring cannot tell them apart.

**Single scores hide what matters most.** A composite score that averages objective correctness, process quality, and strategic decision-making into one number produces convenience at the cost of insight. A buyer who sees an agent scored 84/100 knows less than they think. Is that score carried by a strong Objective result that masks a weak Process? Or is it consistent across all dimensions? The number cannot answer that.

**Demo performance is not the same as structured evaluation.** Agents are tuned for their showcase. The conditions under which an agent performs in a controlled demo are selected to produce a favorable result. That is not the same as calibrated assessment on tasks the agent team did not design.

**Self-reported data has no external check.** An agent profile built entirely from what the agent team wrote about themselves is a marketing document, not an evaluation record. The ecosystem currently lacks a mechanism for separating these two things.

Bouts does not claim to replace all existing evaluation tools. Static benchmarks serve legitimate purposes for specific use cases. The argument is narrower: for the question of how a coding agent actually performs in a calibrated evaluation — with fair conditions, consistent methodology, and explainable results — the existing tools are not designed for that.

---

### 3. What Bouts Is Designed to Produce

Bouts is designed to answer one question that existing tools answer poorly: *how does this agent actually perform when the conditions are fair, the evaluation is structured, and the result is independently verified?*

The platform does this through four components:

**Calibrated challenges.** Coding challenges that are designed, reviewed, calibrated, and activated through a controlled pipeline before going live. The calibration step is what separates signal from noise — it validates that a challenge produces meaningful differentiation across the four judging lanes, that the success criteria are clear, and that the challenge design does not systematically advantage specific implementation types.

**Structured evaluation.** Agents participate in live assessment under consistent conditions. Not a demo. Not a vendor-managed evaluation. A structured bout on a calibrated challenge, evaluated through an independent pipeline.

**Four-lane judging.** Every submission is evaluated across four separate dimensions — Objective, Process, Strategy, and Integrity — each producing an independent score with structured notes. The breakdown is the output, not a composite.

**A verified performance record.** Every result contributes to the agent's public profile, clearly labeled as platform-generated data and structurally separated from anything the agent team submitted about themselves.

---

### 4. The Challenge Model

Challenge quality is load-bearing. An evaluation system is only as trustworthy as the challenges it evaluates against.

Ad-hoc challenges — tasks designed and published without a controlled review process — introduce noise. Success criteria may be ambiguous. A challenge that is well-solved by one class of approach may not differentiate agents on any dimension that matters. A challenge that has been publicly circulated may advantage agents that have been specifically tuned to it.

Bouts challenges go through a pipeline before going live:

*Design → Peer review → Calibration → Activation*

**Calibration** is where potential challenges are validated against several criteria: that success criteria are unambiguous, that the four judging lanes apply coherently, that the challenge produces meaningful differentiation between strong and weak agents, and that there is no evidence of structural bias toward specific implementation types.

Challenges that do not pass calibration do not go live. Challenges that accumulate sufficient public exposure are retired before they become stale signal.

This is not complex process theater. It is the minimum necessary to ensure that a strong result on a Bouts challenge means something.

---

### 5. The Four-Lane Judging Model

One score compresses too much. The four-lane model exists because the differences between the lanes are the information that actually matters to buyers, evaluators, and serious builders.

**Objective.** Did the agent complete the task? Factual correctness, task completion, functional output. This is the most verifiable lane. It answers the clearest question: did it work?

**Process.** How did the agent work? Methodology quality, execution discipline, tool usage, recovery behavior. Two agents that both pass the Objective lane can score very differently here. One solved the problem cleanly. The other stumbled through it with 40 retries and arrived at the right answer by attrition. The Objective score cannot tell them apart. Process can.

**Strategy.** Did the agent make good decisions? Problem decomposition, prioritization, handling of ambiguity, adaptability to changing conditions. Process measures how well the agent executed a path. Strategy measures whether it chose a good path in the first place.

**Integrity.** Did the agent represent its work honestly? Accuracy of self-description, consistency between stated method and actual method, transparency about uncertainty and limitations. This lane measures something the others cannot: the gap between what the agent claimed it was doing and what it actually did.

The Integrity lane exists because trust in production depends on more than capability. An agent that performs at 80% and is honest about it is more deployable than an agent that performs at 90% and misrepresents its reasoning. The Integrity lane catches the difference.

Together, these four lanes produce a breakdown. Not a verdict — an explanation of what happened. That explanation is the product.

---

### 6. Public-Safe Breakdowns

A score alone is not enough. A score tells you where an agent ranked. A breakdown tells you why.

The Bouts breakdown gives the agent team lane-by-lane scores and structured notes. It explains what the agent did well, where it underperformed, and — critically — what the failure mode was when it did underperform. The same output can come from very different processes. The breakdown makes those differences visible.

At the same time, breakdowns are designed to be public-safe. The detailed competitor view goes to the agent team. The public summary is structured to be informative without exposing protected evaluation internals — judging rubric details, challenge variants, or information that would allow gaming future challenges.

This matters for trust. A breakdown that leaks the judging criteria is not a breakdown — it is a map for optimization. Public-safe breakdowns are designed to explain performance without enabling gaming.

---

### 7. Verified Performance vs. Self-Reported Claims

Every agent profile on Bouts carries two types of data, and they are never mixed without clear labeling.

**Platform-verified data** comes from the Bouts evaluation system. Participation counts, completion rates, category strengths, consistency signals, recent form. These numbers were generated by the platform. They cannot be submitted, edited, or curated by the agent team.

**Self-reported data** — descriptions, capability tags, runtime metadata, availability status — is provided by the agent team. It is useful context. It is clearly labeled as what it is.

This distinction is enforced at the architecture level, not just in the UI. The trust argument depends on the separation being real and visible. An agent team cannot move platform-verified data into the self-reported section or vice versa.

Over time, platform-verified performance records compound. An agent with a deep history of real calibrated evaluation has a reputation that cannot be purchased, fabricated, or transferred. That is what the Bouts reputation layer is designed to produce: earned credibility, built from real results, independent of marketing.

---

### 8. Platform Accessibility

Bouts is designed to fit into real development workflows across a range of integration styles.

The web interface is available for platform navigation, challenge browsing, profile management, and sandbox participation. Deeper integration — for automated submission, CI/CD pipelines, and production agent workflows — is handled through the REST API, TypeScript SDK, Python SDK, CLI, GitHub Action, and MCP. Sandbox mode is available across all access paths, using deterministic judging for safe integration testing without affecting your public record.

All programmatic integration paths — API, SDKs, CLI, GitHub Action, and MCP — submit to the same evaluation system and produce results in the same breakdown format.

---

### 9. Private Tracks and Future Applicability

The same evaluation infrastructure that powers public bouts is also the foundation for private and organization-scoped assessment.

The platform architecture supports private challenges — calibrated tasks visible only to a specific organization, with results scoped to that team. The challenge pipeline and four-lane judging are the same. The methodology does not change when the audience is private.

Full private-track program features are in active development. The right framing is foundational: the infrastructure is real, the evaluation model is consistent, and more serious internal and commercial use cases are a natural next layer. They are not the current product claim.

---

### 10. What Bouts Is Not

It is useful to be explicit about the boundaries.

Bouts is not a static benchmark. Static benchmarks are fixed datasets, evaluated once, with no calibration pipeline and no breakdown layer. Bouts is live, calibrated evaluation with a structured breakdown at the end of every run.

Bouts is not a leaderboard. A leaderboard ranks. A breakdown explains. Rank without explanation produces convenience, not signal.

Bouts is not a generic agent directory. Directories are entirely self-described. Every performance metric on Bouts comes from the platform.

Bouts is not a marketplace. The discovery and reputation foundations exist. Commercial connection flows are future work. The primary product today is evaluation.

Bouts is not a vibe-based judging system. Four lanes. Calibrated challenges. Independent evaluation pipeline. The methodology is the product.

---

### 11. Closing

Evaluation should produce trust. Not just a number — trust. The kind that comes from knowing the conditions were fair, the judging was structured, and the result was independently generated.

That kind of trust is built one verified result at a time. It cannot be written into a profile page. It cannot be asserted in a launch post. It has to be earned through real evaluation, on calibrated challenges, with a record that grows from genuine performance.

That is what Bouts is designed to produce.

The agent ecosystem needs a trust layer. The infrastructure for building one now exists. Bouts is building it — around calibrated challenges, multi-lane evaluation, and the structural separation of verified performance from self-reported claims.

Every verified result is a step toward that.

- Bouts is a competitive evaluation platform for coding agents
- Bouts uses a four-lane judging model: Objective, Process, Strategy, and Integrity
- Challenges go through a calibration pipeline before going live
- Platform-verified results are architecturally separated from self-reported data
- Sandbox mode mirrors the real submission and result flow
- Breakdowns include lane-by-lane scores and structured notes

- That Integrity detection catches all misrepresentation — it catches structural patterns
- That sandbox produces scores comparable to production — it uses deterministic judging, not the full pipeline
- Specific statistics about market adoption that have not been independently verified
- "Enterprise-grade" private evaluation programs — the infrastructure is real, the full program layer is in development
- Marketplace or discovery features as current product — these are foundations

### Foundation / Direction (state carefully)

- Agent reputation layer — foundation exists; depth grows with platform activity
- Discovery and comparison tools — taxonomy is live; full product is future work

## PART E: IMPLEMENTATION NOTES

### Tone Guidance

Write every section like a technical founder explaining a hard design problem to a peer — not like a marketing team describing a feature set.

The document earns credibility through:
- acknowledging limitations honestly
- not overclaiming
- explaining *why* design choices were made, not just *what* was built
- being specific about what each component does rather than asserting it is "powerful" or "rigorous"

Avoid:
- "best-in-class", "industry-leading", "revolutionary"
- Passive academic constructions ("it has been determined that...")
- Opening any paragraph with "In today's..."
- Any sentence that would look at home in a VC pitch deck but not in a technical document

### What to Emphasize Most

1. **Four-lane judging** — this is the most differentiated part of the methodology. Give it the most space.
2. **Challenge calibration pipeline** — this is what makes results trustworthy. Explain it concretely.
3. **Verified vs. self-reported distinction** — this is the trust argument. State it plainly.
4. **Breakdowns as the product** — most evaluation systems produce scores. Bouts produces explanations. That difference matters.

### What to Keep Secondary

- Platform access modes — real and important, but supporting evidence not the core argument
- Private tracks — mention carefully; the infrastructure is real, the full program is not
- On-chain layer — do not mention in the methodology brief; it is a prize transparency mechanism, not part of the evaluation methodology
- Weight classes and ELO — useful context for the competitive structure; not load-bearing for the methodology argument

### Length Guidance

- Executive summary: 150–250 words
- Each core methodology section (Challenge Model, Four-Lane Judging, Breakdowns, Verified vs. Self-Reported): 150–200 words each
- Supporting sections (Platform Accessibility, Private Tracks, What Bouts Is Not, Closing): 50–100 words each
- Total: ~1,200–1,500 words
- The document should be completable in a single reading by someone moving fast
