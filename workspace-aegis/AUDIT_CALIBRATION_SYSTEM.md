# Aegis Audit — Challenge Calibration System
**Date:** 2026-04-01
**Auditor:** Aegis 🛡
**Scope:** Full calibration pipeline, admin UX, quality assessment, auto-trigger, integration
**Verdict:** ⚠️ LAUNCH-READY WITH NON-OPTIONAL FOLLOW-UPS

---

## Executive Verdict

**Launch-ready with non-optional follow-ups.**

The calibration system is architecturally sound and the Stage 1 LLM analysis quality is genuinely impressive — challenge-specific, evidence-grounded, and useful for reducing Nick's manual burden. However, there are two trust-breaking issues that must be fixed before relying on this system in production:

1. **P1: Approve/quarantine/adjust wipe the full dossier** — once a reviewer acts on a challenge, the historical calibration evidence is destroyed. This is not cosmetic.
2. **P1: auto-trigger endpoint is unauthenticated via a trust header** — any internet caller can force LLM analysis rewrites on real challenges, incurring cost and resetting reviewer status.

Additionally, there are two pipeline architecture findings that create operational confusion:

3. **P2: Two disconnected calibration pipelines coexist** — Gauntlet cron uses the old `/internal/auto-calibrate` (no dossier, no queue) while the intake route uses the new dossier-based system. Challenges from the old path appear "passed" without ever entering the admin review queue.
4. **P2: `auto_pass` is structurally impossible via auto-trigger** — the decision policy requires a benchmark run for `auto_pass`, but auto-trigger only runs Stage 1. Existing `auto_pass` dossiers in DB were generated by an older code version without this requirement.

## Findings by Severity

### P1 — Major

**AEG-P1-001: Approve/Adjust/Quarantine wipe calibration dossier**
- **Surface:** Admin workflow / dossier integrity
- **Route:** `POST /api/admin/calibration/pipeline` (actions: approve, adjust, quarantine)
- **Root cause:** All three reviewer actions upsert `dossier: {}` — an empty object — overwriting the full calibration evidence stored in the `dossier` jsonb column.
- **Evidence:**
  ```
  # Lines 70-75, 93-97, 114-118 of pipeline/route.ts:
  await supabase.from('challenge_calibration_dossiers').upsert({
    challenge_id,
    dossier: {},   ← DESTROYS the full CalibrationDossier jsonb
    reviewer_status: 'approved',
    ...
  })
  ```
- **Impact:** After approval, the dossier column is `{}`. The admin UI loses: decision rules, per-dimension rationale, reviewer flags, analyzer rationale, exploit vectors, failure modes, hidden constraints. DB confirms: 0 approved dossiers with non-empty content.
- **Confirmed in DB:** No approved dossiers exist yet (no reviewer actions taken), but the bug is live and will destroy evidence on first approval.
- **Fix:** Upsert should preserve the existing dossier and only update reviewer-specific fields: `reviewer_status`, `reviewed_at`, `final_profile`, `reviewer_notes`. Do NOT include `dossier` field in approve/adjust/quarantine upserts.

**AEG-P1-002: auto-trigger endpoint bypasses auth via trust header**
- **Surface:** Internal endpoint security
- **Route:** `POST /api/admin/calibration/auto-trigger`
- **Root cause:** The endpoint accepts `x-forge-internal: true` as sufficient auth — no secret required.
  ```ts
  const isInternal = (cronSecret && authHeader === cronSecret) ||
    req.headers.get('x-forge-internal') === 'true'  // ← anyone can set this
  ```
- **Live test confirmed:**
  ```bash
  curl -X POST https://agent-arena-roan.vercel.app/api/admin/calibration/auto-trigger \
    -H "x-forge-internal: true" \
    -d '{"challenge_id": "00000000-0000-0000-0000-000000000000"}'
  # → HTTP 200, {"skipped":true,"reason":"no prompt or not found"}
  ```
- **Impact with real challenge_id:**
  - Forces an OpenRouter LLM call (Sonnet 4.6) — cost amplification
  - Overwrites the dossier of any analyzed challenge
  - Resets `calibration_reviewer_status` back to `recommended` — can undo a reviewer's approved status
  - Attacker needs only to know a valid challenge UUID (enumerable from public challenge list)
- **Severity note:** Not a data exfiltration risk — only a cost + dossier-tampering risk. Elevated to P1 because it can silently corrupt approved calibration state.
- **Fix:** Remove the `x-forge-internal` fallback. Require `CRON_SECRET` exclusively. The intake route should be updated to use the service-key mechanism or CRON_SECRET.

### P2 — Important

**AEG-P2-001: Two disconnected calibration pipelines with no operator visibility**
- **Surface:** Gauntlet integration / auto-trigger
- **Detail:** Two separate systems process challenges:
  - **Path A (Gauntlet cron):** `POST /api/cron/gauntlet` → direct DB insert → `POST /api/internal/auto-calibrate` → real LLM calibration → auto-promote to `calibration_status=passed` with no dossier, no admin queue entry
  - **Path B (Intake/Admin):** `POST /api/challenges/intake` → `POST /api/admin/calibration/auto-trigger` → Stage 1 LLM analysis → dossier created → challenge appears in admin queue as `recommended`
- **DB evidence:** 6 challenges with `calibration_status=passed` AND `calibration_reviewer_status=unreviewed` — these passed real LLM calibration automatically without any human review or dossier.
- **Impact:** Nick cannot see these challenges in the admin calibration queue (they show as `unreviewed` with `calibration_status=passed`). They bypassed the whole new system. The "Launch Candidates" filter won't surface them properly.
- **Fix:** Gauntlet cron should be updated to also trigger the new dossier pipeline (`/api/admin/calibration/auto-trigger` or equivalent) after auto-calibrate runs, OR migrate the cron to use the intake route.

**AEG-P2-002: `auto_pass` structurally impossible via auto-trigger (code vs live data mismatch)**
- **Surface:** Calibration quality / trust
- **Detail:** Current `decision-policy.ts` requires `AUTOPASS_BENCHMARK_RUN` (benchmark must have been executed) before `auto_pass` is achievable. But `auto-trigger` only runs Stage 1 (no benchmark). Result: auto-trigger will never produce `auto_pass` — always `needs_light_review` minimum.
- **DB evidence:** 5 dossiers show `recommendation=auto_pass` but none have a `decision` field in the dossier jsonb — they were generated by an earlier code version that did NOT require benchmark for auto_pass.
- **Impact:** Operators see `auto_pass` recommendations that can never be reproduced by re-running analysis. If Nick re-analyzes a challenge currently showing `auto_pass`, it will come back as `needs_light_review`. This is confusing and erodes trust in the system.
- **Fix:** Either (a) provide a clear path to run benchmark that unlocks `auto_pass`, or (b) reconsider whether `auto_pass` without benchmark is acceptable. The current UI shows an informational flag ("run benchmark to unlock auto_pass") which is correct but the stale DB data is misleading.

**AEG-P2-003: Synthetic runner uses Math.random() — non-deterministic calibration results**
- **Surface:** Calibration quality / reproducibility
- **Detail:** `synthetic-runner.ts` includes `Math.random()` jitter in score computation (line 192) and a probabilistic `pass_rate` check (line 302). Running the same challenge twice will produce different tier scores and potentially different verdicts (pass vs borderline).
- **Impact:** Borderline challenges can flip between `pass` and `borderline` between runs without any change to the challenge. The benchmark is not reproducible. This is acceptable for a fast pre-filter, but should be clearly disclosed in the admin UI — it isn't.
- **Fix (P2):** Add a note in the admin UI that synthetic benchmark results have variance. Consider seeding with challenge_id hash for reproducibility.

### P3 — Hygiene

**AEG-P3-001: Decision rules not stored for challenges analyzed via run-batch (auto-trigger path)**
- The `dossier` jsonb stored by run-batch and auto-trigger does not include the `decision` field — only the top-level recommendation/confidence/flags are stored. The decision rules (which rules fired, which blocked) are absent from the DB.
- Impact: Admin UI's "Decision Rules" section is empty for all auto-trigger-analyzed challenges. Only pipeline-analyzed challenges show rule traces.
- Fix: Auto-trigger and run-batch should store the full dossier object including `decision` and `per_dimension_rationale`.

**AEG-P3-002: Dossier data/evidence distinction not explicit in UI**
- The UI doesn't clearly label which values are predicted vs measured. The solve rate band, exploitability, and dimension scores are AI predictions presented without a disclaimer that no actual benchmark has been run.
- For the 5 `auto_pass` challenges (old code), the `reviewer_flags` shows "Exploit vectors identified: N" but doesn't clarify this is predicted, not measured.
- Fix (minor): Add a UI callout: "This analysis is Stage 1 only — predictions based on prompt content. Run benchmark for empirical data."

**AEG-P3-003: Adjust action requires non-empty override reason but no server-side validation**
- The UI requires a reason for adjust, but the API (`pipeline/route.ts`) doesn't validate that `reviewer_notes` is non-empty for `adjust` action. A direct API call can adjust a profile without any audit trail.
- Fix: Add server-side validation: `adjust` without `reviewer_notes` should return 400.

## Calibration Quality Assessment

### Is the analysis genuinely challenge-specific?

**Yes — substantially.** This is the strongest part of the system.

**Quarantine example (rate-limiter challenge):** The analyzer correctly identified that no actual buggy code was provided, that agents would fabricate a plausible-sounding fix without real debugging, and listed 4 specific exploit vectors including "without a reference implementation, graders cannot verify the bug was real." This is challenge-specific reasoning, not a template output. The quarantine recommendation is correct.

**Auto_pass example (run-length encoder):** The analyzer correctly identified the multi-digit decode edge case (`a12` as 12 'a's not 'a'+'1'+'2'), the round-trip property preventing hardcoding, and the 6 specific failure modes. Confidence 0.93. Solve rate 82-95%. Recommendation: auto_pass (under old policy) is reasonable.

**Light-review example (stateful Python bug):** The analyzer correctly noted the prompt's own Bug 2 description is internally contradictory, assigned deception=6, and identified the specific regression path where agents fix Bug 1 and 3 but introduce a new regression on Bug 2. Confidence 0.85.

### Are the 8-dimension profiles plausible?

**Mostly yes, with some systematic bias.** 

Observed patterns:
- `reasoning_depth` and `ambiguity` are generally well-calibrated
- `deception` is correctly elevated when prompts contain traps or contradictions
- `time_pressure` appears slightly conservative — many challenges assigned 2/10 even for tight windows
- `evaluation_strictness` of 8/10 for the encoder challenge is accurate (binary round-trip correctness)
- The profiles are NOT generic defaults — they differ meaningfully between challenges

### Are solve-rate and exploitability predictions plausible?

**Plausible but uncalibrated against actual data.** The predictions are directionally correct (easy challenges get high solve rates, high-exploitation challenges are flagged) but the absolute values are estimates. There is no feedback loop yet from actual competition results to validate predictions.

The model correctly identifies:
- High-exploitability: "no actual code provided, agent can fabricate plausible answer" ✓
- Low-exploitability: "round-trip property with dynamic inputs prevents hardcoding" ✓
- Solve rate 82-95% for easy well-specified algorithms ✓

### Is this generic or challenge-aware?

**Genuinely challenge-aware.** The failure modes, exploit vectors, and hidden constraints sections all contain challenge-specific content that could not be generated by a template. The analyzer reads the actual prompt and reasons about it.

**One weakness:** Session length recommendations feel slightly heuristic. 15 minutes for the rate-limiter (no actual code), 20 minutes for the encoder, 2-hour window for encoder. These are plausible but not obviously derived from anything other than the time_limit field and format.

## Auto-Trigger / Gauntlet Integration

### Is calibration auto-triggered when Gauntlet creates a challenge?

**Yes — but through the OLD pipeline, not the new one.**

The Gauntlet cron (`/api/cron/gauntlet`) inserts challenges directly and calls `/api/internal/auto-calibrate`, which runs real LLM calibration and auto-promotes challenges to `calibration_status=passed` without creating a dossier or entering the admin queue.

The intake route (`/api/challenges/intake`) — used when Gauntlet sends bundles — DOES fire the new auto-trigger and creates dossiers.

**Which path Gauntlet uses depends on whether it sends via the intake API or the cron generates directly.** Currently both paths are live.

### Does the reviewer queue catch those challenges?

**Challenges from the Gauntlet cron do NOT appear in the admin calibration queue** (calibration_reviewer_status=unreviewed, no dossier). They are silently marked passed.

Challenges from the intake route DO appear in the queue with dossiers.

### Does the system reduce manual work?

**For intake-sourced challenges: yes, substantially.** The queue surfaces analysis, flags exploitability, pre-rates difficulty. A reviewer can approve or quarantine in seconds with evidence.

**For Gauntlet-cron-sourced challenges: no change.** They bypass the queue entirely.

## Operational Truth Assessment

### Can Nick rely on this without being the calibrator?

**For well-specified, moderate-complexity challenges: yes, with light oversight.**

The system correctly:
- Quarantines obviously broken challenges (no artifact to debug = high exploit)
- Identifies exploit vectors with specificity
- Catches ambiguity issues
- Recommends appropriate session lengths for most challenge types

The system will need Nick for:
- **Challenges where the exploitability call is marginal** (medium vs low boundary)
- **Any challenge getting `needs_deep_review`** — the system is specifically flagging these as needing human judgment
- **Novel challenge formats** where the analyzer may lack reference patterns
- **Challenges where solve-rate band is very wide** (e.g., 30-80%) — the system is uncertain

### Is `auto_pass / light_review / deep_review / quarantine` credible?

**Structurally yes, with one caveat:** `auto_pass` currently requires a benchmark run that the auto-trigger doesn't perform. So in practice:
- Challenges analyzed via auto-trigger land at `needs_light_review` (best case, if confidence high, exploit low, ambiguity low)
- Only `needs_light_review` and above are realistic outcomes from auto-trigger
- `auto_pass` requires manually running the benchmark via the admin UI

This means **every challenge requires at least a light review pass** by someone. That's still substantially less work than full manual calibration, but it's not "automated" in the fullest sense.

### Is the admin workflow good enough for launch?

**Yes, with the two P1 bugs fixed.** Once approve/quarantine/adjust stop wiping dossiers, and auto-trigger is properly secured, the workflow is usable and informative. The UX is well-executed.

### What should still be improved before relying heavily?

1. Fix P1-001: Preserve dossier on approve/adjust/quarantine
2. Fix P1-002: Remove `x-forge-internal` bypass from auto-trigger  
3. Reconcile the two calibration pipelines (Gauntlet cron vs intake)
4. Store `decision` and `per_dimension_rationale` in auto-trigger/run-batch path
5. Add live metrics feedback loop to validate predicted solve rates
6. Consider seeding synthetic runner for reproducibility
