## Judging System Canon (BOUTS_JUDGING_SYSTEM_v1 — confirmed aligned)

4-lane architecture, all confirmed matching live site:
- **Objective** — 50% (published as 45–65% band). Deterministic harness, hidden + visible tests, correctness anchor.
- **Process** — 20%. Telemetry-driven: tool discipline, recovery, iteration quality.
- **Strategy** — 20%. Decomposition, prioritization, ambiguity handling, false-lead rejection.
- **Integrity** — 10% base + asymmetric: up to +10 bonus, down to −25 penalty. Trust/anti-exploit layer.

Judge diversity: Process=Family A, Strategy=Family B, Integrity=Family C, Audit=Family D (standby).
DisputeFlagged: fires when judge spread >15pts, holds reward release, routes to Audit judge.
Tie-break order: Objective → hidden test performance → Process → Strategy → Integrity → time (last resort).

Calibration system + telemetry-driven rebalancing = backend gaps not yet built (Gauntlet work).

## Challenge Quality Automation (Sections 10–11 of BOUTS_JUDGING_SYSTEM_v1)
Built and deployed after Nick confirmed alignment with the full judging spec.

### DB Changes (migration 00020)
New columns on `challenges`:
- solve_rate, score_mean, score_stddev, dispute_rate, exploit_rate, tier_separation
- last_calculated_at, quarantine_reason, quarantined_at, min_required_samples
- calibration_status constraint expanded: draft → calibrating → passed → active → flagged → quarantined → retired → archived

New tables:
- `challenge_quality_snapshots` — historical CDI snapshots for trending/analytics
- `challenge_admin_actions` — full audit trail on every admin override (actor, action, reason, before/after status)

### DB Functions (all SECURITY DEFINER, live in Supabase)
- `compute_challenge_quality(challenge_id)` — calculates solve_rate, score_mean, score_stddev, dispute_rate, exploit_rate, tier_separation, CDI score. Writes back to challenges + inserts snapshot.
- `evaluate_challenge_status(challenge_id)` — applies flag/quarantine thresholds, updates calibration_status, returns reason.
- `can_activate_challenge(challenge_id)` — activation gate. Returns {allowed, reason}. Blocks if calibration_status != passed.
- `run_quality_enforcement_pass()` — runs compute + evaluate on all active/calibrating challenges. Called by cron.

### CDI Formula
CDI = 0.30 * solve_rate_score + 0.25 * score_variance_score + 0.20 * judge_agreement_score + 0.25 * tier_separation_score

### Thresholds (MVP defaults, tune with real data)
Flag if: solve_rate >85% or <5%, score_stddev <8, dispute_rate >12%, exploit_rate >3%, tier_separation <10pts
Quarantine if: exploit_rate >8%, dispute_rate >20%
Min samples before enforcement: 20 runs

### API Routes
- `GET/POST /api/admin/challenge-quality` — admin control: recalculate, flag, unflag, quarantine, unquarantine, retire, force_activate, enforcement_pass. All actions logged to challenge_admin_actions.
- `GET /api/cron/challenge-quality` — cron endpoint for enforcement pass

### Cron

## Hybrid Calibration System (deployed ~10:12 AM KL)

### Files shipped
- `src/lib/calibration/types.ts` — shared interface, CalibrationRunner, TierCalibrationResult, CalibrationResult, MutationType, policy map, thresholds
- `src/lib/calibration/synthetic-runner.ts` — SyntheticCalibrationRunner. Deterministic tier profiles (naive/standard/strong/elite). Difficulty modifier applied. No API calls. Fast + free.
- `src/lib/calibration/real-llm-runner.ts` — RealLLMCalibrationRunner. 4 different model families: Mistral-7B (naive), Llama-3-8B (standard), Gemini Flash (strong), Claude-3.5-Sonnet (elite). Claude-Haiku scores submissions. Fallback to synthetic if any call fails.
- `src/lib/calibration/orchestrator.ts` — routes to synthetic vs real LLM per policy. Conservative verdict (most pessimistic). Persists to challenge_calibration_results, updates calibration_status, logs to challenge_admin_actions.
- `src/lib/calibration/mutation-engine.ts` — semantic/structural/adversarial mutations via Claude-3.5-Sonnet. Returns full challenge payload with updated difficulty_profile, lineage, parent_challenge_id, mutation_generation. Auto-creates new challenge as draft.
- `src/app/api/admin/calibration/route.ts` — admin API: run_synthetic, run_real_llm, run_full, run_forced_real, mutate actions. GET returns all calibration results for a challenge.
- `supabase/migrations/00022_calibration_system.sql` — challenge_calibration_results table

### DB
- New table: `challenge_calibration_results` — stores synthetic + real LLM results per challenge (unique on challenge_id + runner_type)

### Policy map
- daily/special → synthetic only
- standard/versus → synthetic required, real optional (triggers on borderline)
- featured/boss/abyss/prize/versus-stakes → both required

### Thresholds
- Separation ≥ 20pts → pass
- Separation 10–20pts → borderline (triggers real LLM check)
- Separation < 10pts → fail
- Tier spread < 8 → also fail

### What Gauntlet's role now is
Gauntlet's job shifts from "build the infrastructure" to "refine challenge intelligence" — improving the tier profiles, mutation quality, and calibration accuracy over time. The backbone is done.
