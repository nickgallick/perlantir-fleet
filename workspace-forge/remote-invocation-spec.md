## 1. Executive Summary

Remote Agent Invocation (RAI) replaces the manual browser text-submission path with a trust-grounded, browser-convenient submission model. Instead of a user pasting text into a textarea, Bouts sends a signed HTTP request server-side to the user's already-running agent endpoint, captures the machine-generated response and provenance metadata, and submits it into the standard evaluation pipeline — the same pipeline used by the connector, SDK, and CLI.

**The core proposition:** browser convenience without trust regression. Users get a one-click trigger in the browser. Bouts gets a verifiable machine-originated response with timing, endpoint identity, and immutable provenance. No hosted runtime. No code execution. Bouts calls your service; your service answers.

**What ships in three phases tonight:**
- Phase 1: Backend foundation (DB schema, signing lib, invocation engine, secret management, endpoint registration API, invoke API route)
- Phase 2: Workspace UI replacement + settings configuration panel
- Phase 3: Docs, `remote_invocation_supported` challenge flag, admin observability, quickstart update

## 2. Product Definition

### What It Is
A submission path where Bouts sends a HMAC-signed HTTP POST to the user's registered agent endpoint, captures the response, and routes it through the standard submission pipeline. The user triggers the invocation from the browser with a single button click.

### Who It's For
- Developers who have deployed their agent as an HTTP service (cloud, VPS, local tunnel)
- Users who want browser convenience without the text-paste integrity gap
- Teams running persistent agent services who don't want to install the connector CLI on each machine
- Anyone competing from a machine where they can't run a local process

### When to Use RAI vs Other Paths

| Path | When to Use |
|------|-------------|
| **RAI (this)** | Agent is a deployed HTTP service; want browser-native trigger |
| Connector CLI | Agent runs locally as a process; local machine available |
| GitHub Action | Agent runs in CI; submission triggered by push/workflow |
| REST API / SDK | Programmatic control, bulk submissions, custom orchestration |
| MCP | Cursor/Claude Desktop workflow; LLM-native agent tools |

### Trust Level Provided
**Medium-high.** RAI proves machine origin (a running HTTP endpoint responded) and captures timing provenance. It does not prove the endpoint ran an AI model, cannot verify what code ran behind the endpoint, and cannot prove the model used matches the registered model_name. It is categorically stronger than a text box and comparable to API/SDK submission, with additional endpoint-origin provenance.

### What Provenance RAI Captures
- Endpoint URL (snapshot at invocation time — not the current config)
- Invocation timestamp (request_sent_at) and response timestamp (response_received_at)
- Response latency in milliseconds
- HTTP response status code
- SHA-256 hash of raw response content (before any parsing)
- Optional self-declared execution_metadata (model, framework, runtime_ms, tokens_used) — labeled as unverified
- HMAC signature verification result (did the endpoint see a validly signed request? — inferred from successful response)
- Invocation ID (UUID, included in request, echoed or logged)

### What RAI Does NOT Prove
- That an AI model generated the response
- That the model used is the one registered in the agent profile
- That the endpoint contains no pre-computed lookup tables
- That the response wasn't generated by a human before the invocation
- Infrastructure behind the endpoint

## 3. Trust Model

### What Is Verified by Bouts
1. **Endpoint existence**: an HTTP service responded at the registered URL
2. **Request authenticity (one-way)**: Bouts signed the request with HMAC-SHA256; endpoint can verify this is a genuine Bouts invocation
3. **Response integrity**: SHA-256 hash of response body captured at receipt, stored as immutable artifact
4. **Timing**: exact ms-precision timestamps for request sent and response received
5. **Submission source**: `submission_source = 'remote_invocation'` in the DB — cannot be set to this value by any other path

### What Is Machine-Originated (Evidence Captured)
- HTTP response (not browser paste) — proves a running service answered
- Response latency (timing evidence)
- HTTP headers returned by the endpoint
- Optional `X-Agent-Model`, `X-Agent-Framework` response headers (labeled "self-declared, unverified")
- Optional `execution_metadata` JSON in response body (self-reported runtime context)

### What Remains Outside Bouts' Control
- Whether the endpoint ran an AI model (could be a trivial script or lookup)
- Whether the model matches what's registered
- Infrastructure and code behind the endpoint
- Whether the response was pre-staged before the invocation timestamp
- Whether the user owns/operates the endpoint (could be a proxy to someone else's service)

### Comparison to Other Paths

| Property | Manual Text | RAI | Connector | API/SDK |
|----------|-------------|-----|-----------|---------|
| Machine origin proven | ❌ | ✅ HTTP response | ✅ Process-level | ✅ |
| Timing captured | ❌ | ✅ ms-precision | ✅ | ✅ |
| Response hash (immutable) | ✅ | ✅ | ✅ | ✅ |
| Endpoint identity recorded | ❌ | ✅ URL snapshot | N/A | N/A |
| Model verified | ❌ | ❌ self-declared | ❌ self-declared | ❌ self-declared |
| Process integrity | ❌ None | Partial — endpoint fidelity unknown | High — connector attests runtime | Partial |
| Replay protection | N/A | ✅ timestamp + nonce | ✅ | ✅ idempotency key |
| Signed request | N/A | ✅ HMAC-SHA256 | ✅ | N/A |

**Why RAI is meaningfully stronger than a text box:**
A text box proves only that a human pasted text into a form. RAI proves that a registered HTTP endpoint at a specific URL responded to a Bouts-signed request within a measurable time window. The submission cannot be a copy-paste from another source without also operating an HTTP endpoint that accepts Bouts' signed payload format. This is proof of machine origin — not proof of AI authorship, but a categorical upgrade from textarea trust.

## 4. Endpoint Registration Design

### UI Location
- **Primary:** `/settings` → "Agent" tab → "Remote Invocation" subsection per agent
- **Contextual:** Challenge workspace page — if endpoint not configured, show inline CTA: "Configure your endpoint to use Remote Invocation → [Configure in Settings]"

### Scope
**Per-agent.** Each agent in Bouts has its own endpoint configuration. A user with multiple agents can configure different endpoints for each. Rationale: agent identity is the competitor identity in Bouts — each agent should have its own verifiable endpoint.

### Required Fields
```
endpoint_url: string (https://, max 512 chars)
```

### Optional Fields
```
timeout_ms: integer (10000–120000, default 30000)
max_retries: integer (0–2, default 1)
```

### Secrets / Auth Handling
- On first endpoint configuration, Bouts auto-generates a 32-byte random secret prefixed `rai_`
- Secret is shown **once** in a one-time-reveal modal (same UX as API token creation)
- SHA-256 hash of secret stored in `agents.remote_endpoint_secret_hash` (display confirmation only)
- Plaintext stored in `agent_rai_secrets` table (service-role-only access; RLS `USING (false)` blocks all JWT access)
- User copies the secret and configures it in their endpoint for HMAC verification
- Separate secrets for production and sandbox endpoints

### Endpoint Validation
- URL validation on save: must be `https://`, valid hostname, max 512 chars
- `localhost` / `127.0.0.1` / `10.x.x.x` / `192.168.x.x` / `169.254.x.x` private-range IPs blocked in production (SSRF protection)
- No DNS pre-resolution on save — validation is format-only at registration time

### Test Ping / Health Check
- Explicit "Test Connection" button in settings UI
- Sends a minimal signed POST with `{"type": "ping"}` payload
- Any 2xx response = ping success
- Records `remote_endpoint_last_ping_at` and `remote_endpoint_last_ping_status` (`ok` | `timeout` | `error`) on agent
- Ping is advisory — a passing ping does not gate challenge invocation; a failing ping shows a warning

### Production vs Sandbox Endpoint Configs
- **Separate endpoint URLs and secrets** for production vs sandbox
- Production endpoint: used when invoking on active, non-sandbox challenges
- Sandbox endpoint: used when invoking on sandbox challenges (is_sandbox=true)
- `X-Bouts-Environment: production | sandbox` header sent in every request
- If sandbox endpoint not configured and user triggers on a sandbox challenge → Bouts falls back to showing "sandbox endpoint not configured" state with option to use production endpoint or configure separately
- This separation prevents production traffic from accidentally hitting dev/test endpoints

## 5. Invocation Contract

### Request — What Bouts Sends to the Endpoint

```
POST {endpoint_url}
Content-Type: application/json
X-Bouts-Signature: sha256={hmac_hex}
X-Bouts-Timestamp: {unix_ms}
X-Bouts-Nonce: {16_byte_hex}
X-Bouts-Environment: production | sandbox
Idempotency-Key: {64_hex_chars}
```

```json
{
  "schema_version": "1.0",
  "invocation_id": "uuid-v4",
  "challenge_id": "uuid",
  "session_id": "uuid",
  "entry_id": "uuid",
  "agent_id": "uuid",
  "challenge": {
    "title": "string",
    "prompt": "string | null",
    "format": "string (sprint|standard|marathon)",
    "weight_class": "string (lightweight|middleweight|heavyweight|frontier)",
    "time_limit_seconds": "number | null",
    "expected_output_format": "text | json | code"
  },
  "submission_deadline_utc": "ISO8601",
  "environment": "production | sandbox",
  "idempotency_key": "64_hex_chars"
}
```

**Signature payload construction:**
```
method + "\n" + url + "\n" + timestamp_ms + "\n" + nonce + "\n" + sha256(body)
X-Bouts-Signature = "sha256=" + HMAC-SHA256(secret, payload).hex()
```

### Response — What the Endpoint Must Return

**Success (HTTP 200):**
```json
{
  "content": "string (required — submission body, max 100KB UTF-8)",
  "execution_metadata": {
    "model": "string (optional, self-declared)",
    "framework": "string (optional)",
    "runtime_ms": "number (optional)",
    "tokens_used": "number (optional)"
  },
  "agent_version": "string (optional, semver)",
  "notes": "string (optional, max 1KB — agent self-annotation)"
}
```

**Error (HTTP 4xx / 5xx):**
```json
{
  "error": "string",
  "retryable": "boolean (optional — hint to Bouts retry logic)"
}
```

### Timeout Behavior
- If endpoint does not return a complete response within `timeout_ms`: abort, `outcome = 'timeout'`, no retry
- Partial response (connection drop mid-stream): `outcome = 'error'`, retry eligible

## 6. Auth / Signing Design

### HMAC-SHA256 Request Signing

Every request Bouts sends is signed with the per-agent secret:

```
signature_payload = METHOD + "\n" + URL + "\n" + timestamp_ms + "\n" + nonce + "\n" + SHA256(body_bytes)
X-Bouts-Signature = "sha256=" + hex(HMAC-SHA256(plaintext_secret, signature_payload))
```

- `timestamp_ms`: Unix milliseconds (UTC), sent as `X-Bouts-Timestamp` header
- `nonce`: 16-byte cryptographically random hex, sent as `X-Bouts-Nonce` header — unique per invocation
- `SHA256(body_bytes)`: hex digest of raw request body bytes (before any JSON parsing)
- `plaintext_secret`: fetched from `agent_rai_secrets` table at invocation time, never logged, never in client code

### Per-Agent Secrets
- Every registered endpoint gets its own secret (not shared across agents or users)
- Production and sandbox endpoints have separate secrets
- No global Bouts signing key — compromise of one secret only affects that agent's endpoint

### Replay Protection
- **Timestamp window**: ±5 minutes from Bouts server time. Requests outside window must be rejected.
- **Nonce**: 16-byte random hex, globally unique. Stored in `rai_invocation_nonces` (TTL 10 minutes). Duplicate nonce = reject.
- Endpoint implementations must check both timestamp and nonce to be fully protected.

### Secret Storage
- Plaintext: stored in `agent_rai_secrets` table — RLS `USING (false)` blocks all JWT access; only service_role client can read
- Hash: stored in `agents.remote_endpoint_secret_hash` — shown as prefix in UI for confirmation only
- Never in logs, never in HTTP responses except on creation/rotation (one-time plaintext reveal)
- Never in client-side code — all signing happens server-side

### Secret Rotation
- User can regenerate secret at any time from Settings → Agent → Remote Invocation → "Rotate Secret"
- Old secret is immediately invalidated (Vault entry overwritten)
- New plaintext shown once in modal
- Any in-flight invocations using old secret will fail signature verification on the endpoint — acceptable trade-off
- Recommendation in docs: rotate during low-traffic window

### Verification Guidance for Users (in docs)
```typescript
// Node.js example
import { createHmac, createHash, timingSafeEqual } from 'crypto'

function verifyBoutsRequest(req, secret) {
  const signature = req.headers['x-bouts-signature']
  const timestamp = req.headers['x-bouts-timestamp']
  const nonce = req.headers['x-bouts-nonce']

  // 1. Check timestamp window (±5 minutes)
  if (Math.abs(Date.now() - parseInt(timestamp)) > 300_000) return false

  // 2. Compute expected signature
  const bodyHash = createHash('sha256').update(req.rawBody).digest('hex')
  const payload = ['POST', req.url, timestamp, nonce, bodyHash].join('\n')
  const expected = 'sha256=' + createHmac('sha256', secret).update(payload).digest('hex')

  // 3. Constant-time comparison
  return timingSafeEqual(Buffer.from(signature), Buffer.from(expected))
  // 4. Track and reject duplicate nonces (store nonce, expire after 10min)
}
```

## 7. Timeout / Retry / Failure Behavior

### Max Invocation Duration
- Configurable per agent: 10,000ms – 120,000ms (default 30,000ms)
- Hard server-side cap: 120,000ms (AbortController enforced)
- Note: Vercel serverless timeout is 60s by default — invocation runs in API route, not edge function. For 120s max, route config sets `maxDuration: 120`.

### Retry Logic
- Max retries: 0–2, configured per agent (default: 1)
- Retry delay: 2 seconds fixed (no exponential for v1 — predictable behavior)
- Fresh nonce per retry attempt

### Retryable vs Terminal

| Condition | Retryable |
|-----------|-----------|
| Connection refused (ECONNREFUSED) | ✅ Yes |
| Network timeout (AbortError) | ❌ No — endpoint is too slow |
| HTTP 5xx + `retryable: true` in body | ✅ Yes |
| HTTP 5xx + `retryable: false` or absent | ❌ No |
| HTTP 4xx (any) | ❌ No — endpoint rejected the request |
| Invalid JSON response | ❌ No |
| Content > 100KB | ❌ No |
| Signature-verified successful response | ✅ N/A — success |

### User-Facing Failure States (displayed in workspace + status page)

| `outcome` | User-Facing Message |
|-----------|-------------------|
| `timeout` | "Your agent endpoint did not respond within {X}s. Check that your endpoint is running and reachable." |
| `error` | "Your endpoint returned HTTP {status}. Check your endpoint logs for details." |
| `invalid_response` | "Your endpoint returned a response Bouts couldn't parse. Ensure it returns `{ content: string }`." |
| `content_too_large` | "Your agent's response exceeded the 100KB limit. Trim the output and try again." |

On failure, the entry stays in `workspace_open` status — the user can retry the invocation if the failure was transient. Except on `content_too_large` (no retry until they fix their agent).

### Partial Failure Behavior
- Invocation succeeds + DB write fails: show error, keep entry `workspace_open`, user can retry
- Invocation succeeds + enqueue_judging_job fails: submission recorded as `received`, worker picks it up via dead-letter recovery
- HTTP response received but response body is malformed: logged as `invalid_response`, entry stays open

### HTTP Error Handling
- Never leak raw error messages from the endpoint to the user UI — sanitized to first line, max 200 chars
- Error messages from `agent_rai_secrets` are never returned to client

## 8. Pipeline Integration

RAI is a **single unified pipeline** — not a side channel.

```
User clicks "Invoke Agent" in browser
  → POST /api/challenges/[id]/invoke (authenticated, JWT)
  → [Validate entry ownership + challenge active + endpoint configured]
  → [Resolve session_id from entry]
  → [Build invocation payload from challenge record]
  → [Fetch plaintext secret from agent_rai_secrets (service-role)]
  → [Sign request: HMAC-SHA256]
  → [Claim nonce in rai_invocation_nonces]
  → [HTTP POST to endpoint_url with timeout]
  → [Validate response shape + content size]
  → [Log invocation to rai_invocation_log (provenance record — always written)]
  → On success:
    → [Insert submission (submission_source='remote_invocation')]
    → [storeArtifact() SHA-256 immutable hash]
    → [logSubmissionEvent('received')]
    → [captureVersionSnapshot()]
    → [Update entry status → 'submitted']
    → [enqueue_judging_job()]
    → [logSubmissionEvent('queued')]
    → Return { submission_id, submitted_at, status: 'queued' }
  → On failure:
    → [Log failure to rai_invocation_log]
    → Return { error, outcome, user_message }
    → Entry stays 'workspace_open' (retryable failures)
  → Browser redirects to /submissions/[id]/status (same as web path)
  → Same judging orchestrator (13 stages)
  → Same breakdown generation
  → Same result display
```

No special judging lane for RAI. `submission_source` is metadata — scoring is identical.

## 11. Product Boundaries / Fallback Behavior

### No Endpoint Configured
- Show "Configure Endpoint" CTA in workspace
- No fallback to text submission in production — the manual textarea is removed
- Sandbox / practice path: sandbox challenges MAY retain a "Quick Practice" text path, clearly labeled as sandbox-only and not submittable to production scoring. Spec decision: **do not build this now** — keep it clean and add only if needed.

### Endpoint Unreachable (connection refused, DNS failure)
- `outcome = 'error'`, entry stays `workspace_open`, retry available
- User message: "Your endpoint couldn't be reached. Check that your agent is running."

### Bad Output (invalid response shape)
- `outcome = 'invalid_response'`, entry stays `workspace_open` (can retry after fixing endpoint)
- Exception: if content_too_large, no retry until agent is fixed

### Slow Endpoint
- Entry stays `workspace_open` if timeout reached — user can retry after fixing latency
- Max 3 attempts total (1 initial + 2 retries) per entry per session — not per overall entry lifetime

### Post-Timeout (session expired + endpoint failed)
- If session timer expired before successful invocation: entry moves to `expired`, no further submission
- RAI invocation uses challenge time_limit — user must trigger before session expires

### Challenge Doesn't Support Remote Invocation
- `challenges.remote_invocation_supported = false`
- Workspace shows "This challenge requires connector/API submission"
- Links to connector docs

## 12. Fairness / Methodology Implications

### vs Connector Path
- RAI response is treated identically in all 4 judging lanes
- Process lane evaluates artifact quality (same as connector path)
- Integrity lane records `submission_source = 'remote_invocation'` — this is informational metadata, not a scoring penalty
- A judge receiving a remote-invocation submission cannot distinguish it from a connector submission at the content level

### vs Richer Runtime Path (future)
- RAI has higher latency variance (network roundtrip to user's endpoint)
- `response_latency_ms` is captured but not used in scoring for v1
- When/if process scoring evolves to weight execution efficiency, this can be factored in (it's captured — not yet used)

### Submission Source Visibility
- `submission_source` exposed in competitor breakdown (own only)
- Not in public leaderboard
- Not a scored dimension

### Leaderboard Fairness
- No scoring advantage or penalty based on submission_source
- All sources compete on the same 4-lane scoring model
- RAI does not get "machine origin bonus" — it earns the same score on the same criteria

## 13. Security / Abuse

### SSRF Protection
- **Block private-range IPs and localhost at registration time:**
  - `127.0.0.0/8` (loopback)
  - `10.0.0.0/8` (private)
  - `172.16.0.0/12` (private)
  - `192.168.0.0/16` (private)
  - `169.254.0.0/16` (link-local / AWS metadata)
  - `::1` (IPv6 loopback)
  - `fc00::/7` (IPv6 unique local)
- Validate at registration + at invocation time (DNS rebinding protection: re-resolve and re-check before connecting)
- `https://` required — no `http://`, no `file://`, no other schemes

### Malicious Endpoints
- Response content is treated as opaque text — not executed, not rendered as HTML
- Content stored as artifact and submitted to judging — same pipeline as all submissions
- `execution_metadata` is stored as JSONB but never executed or trusted for scoring
- Max content size enforced (100KB)

### Callback Misuse
- Endpoint URL is registered per-agent, not per-invocation — cannot be changed dynamically
- Invocation ID is Bouts-generated (UUID) — user cannot influence the trace ID

### Endpoint Spam / DoS
- Rate limit on `/api/challenges/[id]/invoke`: 5 invocations per user per hour (generous but bounded)
- Max 3 attempts per entry (1 initial + configured retries, max 2)
- Bouts only invokes on behalf of authenticated users with valid entries — no unauthenticated trigger path

### Secret Leakage
- Plaintext never in logs, never in HTTP responses after creation
- `agent_rai_secrets` table: RLS `USING (false)` — no JWT read path exists
- Service-role client used only in server-side invocation code, never in client components

### Response Poisoning
- Response content is hashed before processing — hash stored as immutable artifact
- Content is not evaluated in any way that could affect Bouts platform behavior
- `notes` field is stored as text — not rendered as HTML anywhere

### Overlong Payloads
- `content` field: 100KB max (same as existing submission limit)
- Full response body: 200KB max before parsing (extra margin to catch malformed payloads before schema validation)
- `notes`: 1KB max

### Private-Network Blocking (DNS Rebinding Defense)
- At invocation time, before sending the HTTP request, resolve the hostname and check the resolved IP against the blocked ranges
- If IP is in blocked range: log `outcome = 'error'`, user message: "Endpoint resolved to a private/reserved IP address and cannot be reached."

### Rate Limiting
- `/api/challenges/[id]/invoke`: 5/hour per user (`invoke:trigger` policy)
- Endpoint registration (PUT/DELETE): 10/hour per user
- Ping: 10/hour per agent

```sql
  response_received_at timestamptz,

  response_latency_ms integer,
  response_content_hash text,
  execution_metadata jsonb,
  attempt_number integer NOT NULL DEFAULT 1,
  outcome text NOT NULL CHECK (outcome IN ('success','timeout','error','invalid_response','content_too_large')),
  error_message text,
  created_at timestamptz NOT NULL DEFAULT now()
);

-- Remote invocation support flag on challenges
ALTER TABLE challenges
  ADD COLUMN remote_invocation_supported boolean NOT NULL DEFAULT true;

-- Extend submission_source constraint
ALTER TABLE submissions
  DROP CONSTRAINT IF EXISTS submissions_submission_source_check;
ALTER TABLE submissions
  ADD CONSTRAINT submissions_submission_source_check
  CHECK (submission_source IN (
    'web','remote_invocation','connector','api','sdk','github_action','mcp','internal'
  ));

-- Nonce cleanup function
CREATE OR REPLACE FUNCTION cleanup_expired_rai_nonces() RETURNS void AS $$
  DELETE FROM rai_invocation_nonces WHERE used_at < now() - interval '10 minutes';
$$ LANGUAGE sql SECURITY DEFINER;
```

### Indexes
- `idx_rai_log_agent_id` — query invocations by agent
- `idx_rai_log_submission_id` — link invocation to submission
- `idx_rai_log_entry_id` — query by entry
- `idx_rai_log_created_at` — time-range queries

## 15. Admin / Observability

### Invocation Log
- `rai_invocation_log` queryable in admin dashboard
- Per-entry invocation history visible in admin submission detail view
- Fields shown: endpoint_url (admin only), outcome, latency, status code, attempt_number

### Failure Debugging
- `outcome` distinguishes endpoint failure (timeout/error/invalid_response) from platform failure (DB error, enqueue failure)
- Platform-side failures logged to submission_events with error details
- Endpoint failures isolated to rai_invocation_log — don't pollute submission_events

### Endpoint Failure vs Platform Failure Distinction
- `rai_invocation_log.outcome` = endpoint's fault (timeout, bad response, 5xx)
- `submission_events` with `event_type='failed'` = platform's fault (DB write, enqueue)
- Admin can tell at a glance: is this the user's problem or ours?

### Dead-Letter Handling
- If invocation succeeds but enqueue_judging_job fails: submission stays `received`, existing dead-job recovery in process-judging-jobs cron picks it up within 2 minutes
- No special dead-letter for RAI — inherits existing recovery

### Admin Challenge Health Tab
- Add "Remote Invocation Failure Rate" metric: % of RAI attempts that failed per challenge (over last 24h)
- Surface challenges with high RAI failure rates (could indicate payload format issue or payload too large)

## 18. What Should NOT Be Built Now

These are explicitly out of scope for this launch. Do not let scope creep introduce any of these:

- **Hosted runtime** — Bouts does not execute user code. Ever. Full stop.
- **Browser IDE / code editor** — not a Monaco editor, not a sandboxed environment, not a terminal
- **File upload** — RAI returns text content only. No binary, no file attachments.
- **Arbitrary code execution** — no Docker, no sandboxed containers, no serverless execution on Bouts infrastructure
- **Agent hosting** — Bouts does not host user agents. Users operate their own endpoints.
- **Giant orchestration layer** — no workflow engine, no step functions, no multi-step invocation chains
- **Streaming responses** — v1 is request/response. No SSE, no WebSocket from endpoint.
- **Multiple submissions per invocation** — one invocation produces at most one submission
- **Invocation history UI** — log is written for admin/debug; no full invocation history page for users (Phase 4 if needed)
- **Sandbox text fallback** — do not add a manual textarea fallback "for practice" — it re-introduces the trust problem we're eliminating
